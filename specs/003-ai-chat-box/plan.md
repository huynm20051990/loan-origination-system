# Implementation Plan: AI Chat Box

**Branch**: `003-ai-chat-box` | **Date**: 2026-04-08 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/003-ai-chat-box/spec.md`

## Summary

Add a persistent sidebar chat panel to the EasyApply home page that accepts natural language
home-search queries. A new `chat-service` microservice (hexagonal, Spring Boot 3.5.x + Spring AI
1.1.2) receives the query, calls `home-service`'s existing `GET /api/v1/homes/search` endpoint,
then streams an AI-generated conversational reply back to the Angular frontend via Server-Sent
Events (SSE). Chat history is session-scoped and stored in Cassandra via the Spring AI
`CassandraChatMemoryRepository`, mirroring the `assessment-service` pattern exactly.

---

## Technical Context

**Language/Version**: Java 21 LTS — virtual threads enabled (`spring.threads.virtual.enabled: true`)
**Primary Dependencies**: Spring Boot 3.5.x, Spring AI 1.1.2 (Google GenAI / Gemini 2.5 Flash,
  `spring-ai-starter-model-chat-memory-repository-cassandra`, `spring-webflux` for SSE),
  MapStruct 1.6.3, Micrometer + Brave tracing, Prometheus
**Storage**: Cassandra (Spring AI chat memory only — no custom schema beyond the keyspace init)
**Testing**: JUnit 5 + Spring Boot Test, Testcontainers (Cassandra), WireMock (home-service stub),
  Jest + Angular Testing Library
**Target Platform**: Kubernetes / Linux, JVM 21
**Project Type**: Web microservice + Angular SPA sidebar component
**Performance Goals**: First SSE token < 3 s (constitution mandate); listing event < 3 s (SC-001)
**Constraints**:
- Submit action disabled while a query is in-flight (FR-008 clarification)
- Cassandra memory managed entirely by Spring AI — no hand-rolled CQL beyond keyspace creation
- SSE requires `spring-webflux` on the classpath alongside `spring-boot-starter-web` (servlet stack
  returns `Flux<ServerSentEvent>` using the Spring MVC + WebFlux co-existence pattern)
- No cross-session persistence; `sessionId` is a UUID generated by the frontend on page load
**Scale/Scope**: Public endpoint — all home page visitors; no RBAC on `/api/v1/chat/**`

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| New microservice (chat-service) | Separates session-scoped chat/AI orchestration from listing-data domain | Adding chat logic to home-service would couple Spring AI chat memory, SSE streaming, and listing persistence — violates single responsibility and prevents independent scaling |
| No JPA / relational DB in chat-service | chat-service stores only chat memory (Cassandra via Spring AI); home listing data is never persisted | Not a violation — explicitly calling out the intentional absence of `spring-boot-starter-data-jpa` and any PostgreSQL dependency to prevent future drift |
